# Память

## Цель

Хранить **весь** контекст о пользователе и диалогах, но в промпт передавать только **релевантный и компактный** срез.

## Три слоя

```
┌──────────────────────────────────────────────────┐
│  Layer 1: User facts (структурированные факты)   │  ← всегда в промпте (~500 токенов)
├──────────────────────────────────────────────────┤
│  Layer 2: Episodes (сжатые эпизоды по темам)     │  ← 2-3 релевантных эпизода (~800 токенов)
├──────────────────────────────────────────────────┤
│  Layer 3: Working memory (последние 8 реплик)    │  ← всегда в промпте (~1000 токенов)
└──────────────────────────────────────────────────┘
```

Итого в промпте: ~2300 токенов контекста памяти + системный промпт.

## Таблицы БД

### `user_facts`
Долговременные факты о пользователе. Ключ уникален — новое значение перезаписывает старое.

| Поле | Тип | Описание |
|---|---|---|
| key | str UNIQUE | например: `occupation`, `goal`, `preference:tone` |
| value | str | например: `строю fintech стартап` |
| source_message | str? | оригинальное сообщение, из которого извлечён факт |
| created_at / updated_at | datetime | |

### `conversations`
Все сообщения (никогда не удаляются). `processed_at IS NULL` → не вошли ни в один эпизод.

| Поле | Тип | Описание |
|---|---|---|
| role | str | `user` или `assistant` |
| content | str | текст сообщения |
| tool_calls | JSONB? | вызовы инструментов (если были) |
| processed_at | datetime? | когда вошло в эпизод |

### `episodes`
Сжатые резюме диалогов. Создаются фоновым заданием суммаризации.

| Поле | Тип | Описание |
|---|---|---|
| summary | TEXT | краткое описание что обсуждалось |
| period_start / period_end | datetime | временной диапазон исходных разговоров |
| topics | TEXT[] | список тем (используется для отбора релевантных эпизодов) |

## Сборка контекста

```python
# memory/store.py
def build_context(message: str, db) -> ContextBundle:
    facts    = get_all_user_facts()
    recent   = get_last_n_conversations(n=8)
    episodes = get_relevant_episodes(
        query_topics=extract_topics(message),
        limit=3,
    )
    return ContextBundle(facts, recent, episodes)
```

Отбор эпизодов — по пересечению тем (SQL `topics && ARRAY[...]`) + recency. Без векторного поиска — достаточно для личного ассистента.

## Фоновая суммаризация

Задание APScheduler запускается каждые `SUMMARIZE_INTERVAL_HOURS` часов (default: 6).

**Условие запуска**: необработанных сообщений старше 24ч не менее `SUMMARIZE_MIN_MESSAGES` (default: 10).

**Модель**: Haiku — дешёвая обработка текста.

**Шаги**:
1. Взять batch необработанных `conversations`
2. Haiku → `{summary, topics}` → сохранить в `episodes`
3. Haiku → новые факты → `upsert` в `user_facts`
4. Проставить `processed_at` у обработанных строк

Сырые `conversations` остаются в БД навсегда — эпизоды это только сжатое представление.
